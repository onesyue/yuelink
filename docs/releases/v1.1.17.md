## Fixes

- **桌面 TUN ↔ 系统代理 切换不再"卡端口"** — 之前在桌面端切换连接模式时，旧 mihomo 进程退出后 `:7890` (mixed-port) / `:1053` (DNS) / external-controller 这几个监听端口没有被干净释放，新核心起来时要么 bind 失败、要么界面看到的出口节点跟实际访问的不是同一个（典型现象：TUN 模式下出口 IP 显示韩国，但点切节点没反应）。本版从 mihomo 内核侧根治：停止流程主动关闭所有 listener + DNS + 控制器，端口立即释放，下一次启动 / 切换之后界面与实际流量完全一致。
- **`allow-lan: true` 配置下端口冲突探测更准** — macOS / BSD 在 IPv6 wildcard 已被占用时仍允许 `bind(127.0.0.1:port)` 表面成功，原本的端口检测会"假阴性"——把已经被旧核心占着的端口判成空闲。本版同时探测 `127.0.0.1` 与 `0.0.0.0`，任一失败即视为占用，端口冲突的根源被堵住。
- **诊断报告新增 mihomo 子进程日志** — 桌面 TUN 模式下，mihomo 是由特权 helper 启动的子进程，日志写在 `mihomo-service.log`；之前导出诊断只包含主进程的 `core.log`，子进程实际行为缺失，排查 TUN 故障要二次往返。本版诊断包同时收两份核心日志。

## Improvements

- **启动时"发现新版本"通知现在可点** — 点一下通知胶囊直接弹出更新对话框（含发版说明 + 下载按钮），不必再绕到「我的 → 检查更新」二次触发（issue #4）。
- **更新下载从单点直连升级到多镜像链** — 之前下载安装包只走 GitHub Release 直连，国内常稳在 50–200 KB/s。本版改为优先走 `gh-proxy.com` / `ghfast.top` 两个 CDN 包装器，原直连作兜底，每条镜像有 15s 连接超时 / 20s 首字节超时 / 15s 无进度看门狗——卡住的镜像自动切换到下一条，国内下载预期提升到 5–20 MB/s。
- **macOS 自动安装新版本** — 下载完 .dmg 后无需手动拖拽。新流程：静默挂载 dmg → 拷贝 YueLink.app 到 /Applications → 清理 macOS 隔离标记 → 卸载 dmg → 重启新版本。挂载 / 拷贝 / 重命名任一步失败都会回退到原"打开 dmg 让用户拖"流程，绝不会把运行中的应用包改坏。
- **DNS 缓存升级到 ARC 算法** — 替换之前的 LRU。代理客户端的 DNS 工作负载里，频繁回访的几十个热点域名 + 长尾一次性查询是典型组合，ARC 在这类负载下命中率比 LRU 高 20–40%。
- **桌面端 `/etc/hosts` 在 fake-IP 模式下也能生效** — 之前桌面端走 fake-IP，自定义 hosts 条目会被绕过；本版默认开 `use-system-hosts: true`，hosts 优先于 fake-IP。

## Upgrade Notes

- 已登录用户无需重登。登录态、订阅、Emby、播放记录、本地设置全部保留。
- macOS 用户：从 v1.1.16 升上来时，如果之前 TUN 切换留下了僵尸 mihomo 进程，本版第一次启动前手动确认下：托盘退出 YueLink → `pkill -9 yuelink-mihomo yuelink-service-helper` → 再启动新版。装上 v1.1.17 之后，后续切换都不会再出现这种残留。
- macOS 用户首次自动更新到 v1.1.17 之后的版本时会走新的"挂载-拷贝-重启"流程；任一步失败会自动回退到老的"打开 dmg 让用户拖"流程，体验等同 v1.1.16。
- 内部把 `core/mihomo` 子模块从 `MetaCubeX/mihomo` 切到自维护 fork `onesyue/mihomo` 的 `main` 分支（上游 Meta + 三个针对 c-archive 嵌入的修复 commit）。普通用户感知不到；从源码 `--recursive` clone 的开发者会拉到新的 fork 仓库。
