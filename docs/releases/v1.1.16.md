## Fixes

- **桌面 TUN 模式切换更稳** — 系统代理 → TUN 来回切换不再卡在「DNS 接管失败」误报；切换后状态判定改为以实际运行的核心为准，不再被界面上的「目标模式」骗到。
- **多 VPN 共存识别更准** — 同时跑 WireGuard / Tailscale / Cisco AnyConnect 等其它 VPN 时，TUN 路由检查只认走全局默认路由（`128.0/1` / `0/1`）的 utun，不再因为机器上有别的 utun 接口就直接判定「TUN 已生效」或「清理失败」。
- **DNS 接管检测改为端到端** — 不再依赖 mihomo 内部的 `/dns/query` 接口判断是否劫持成功（该探针在某些代理节点波动时会假阴性），改为发起一次系统级 DNS 解析、检查返回是否落在 fake-IP 段（`198.18.0.0/15`），更贴近用户真实流量路径。
- **macOS 异常退出后系统 DNS 自动恢复** — 上次 TUN 会话崩溃 / 强杀导致系统 DNS 被遗留为公网解析器（114 / 223 / 8.8.8.8）时，下次正常停止会被识别为残留并改回 DHCP 自动获取，不再永久污染用户原始 DNS 设置。
- **macOS 桌面 TUN 接管 IPv6** — 原本 IPv6 流量绕过 TUN 直接走系统默认 v6 路由会泄漏真实 IPv6 地址，现在 TUN 接口同时分配私有 ULA 并接管 v6 全局路由。

- **断开操作不再被自动恢复反向覆盖** — 心跳自动恢复 / 网络切换重连排在用户「断开」之后执行时，会读取持久化的 manual-stop 标记并主动跳过，不会让核心又自己连回来。
- **热切换连接模式不再误标 manual-stop** — 切换 TUN ↔ 系统代理过程中，停止那一步不再写入 manual-stop 标记，避免极端情况下中途崩溃导致下次启动跳过自动连接。
- **桌面服务模式启动失败的状态更干净** — 服务模式启动失败时，即使诊断报告本身写入异常，`_serviceModeActive` 标记也保证被复位；helper service 显式停止时清零 watchdog 崩溃计数，避免历史累积影响下一次启动。
- **resume 期间不再短路启动** — 从后台恢复时，如果检测到生命周期正在切换 / 重启过程中，跳过状态恢复；避免恢复路径与切换路径竞争把核心标记成「已运行」但其实没真启动。
- **桌面 TUN cleanup 快照不再误报清理失败** — 同上多 utun 共存场景，停止后的 cleanup 检查改为基于路由判定，不再因为别的 VPN 接口存在就把状态写成 `cleanup_failed`。
- **macOS 网络服务列表写入前强制刷新** — 设置 TUN DNS 时先废弃 5 分钟缓存，避免用户中途插了 USB-C 网卡 / 启动了 USB tethering 后新接口的 DNS 没被重写。

## Upgrade Notes

- 已登录用户无需重登。登录态、订阅、Emby、播放记录和本地设置会保留。
- macOS 用户：如果之前 v1.1.15 或更早版本异常退出过，本版第一次正常停止 TUN 时会自动把残留的公网 DNS（114/223/8.8.8.8）改为 DHCP 自动获取。如果你本来手动配的就是这个组合，停止后请到「系统设置 → 网络 → DNS」重新填回。
- macOS 桌面 TUN 现在会接管 IPv6 全局流量。如果你常用的代理节点不支持 IPv6 出向，访问 IPv6-only 站点（少数 Cloudflare 边缘、部分企业网）可能比之前慢一拍（连接失败后回退 IPv4）；大部分双栈站点无感。
