v1.0.21 是 v1.0.20 之后的窄范围 hotfix，专门收口 v1.0.20 用户反馈中复现率最高的几个问题。基于 v1.0.20（`7b2bed8`）之后 9 个 commit，无功能新增，无依赖升级。

### 发布亮点

- **断开后切回不再误显示「已连接」**：用户在 Android 上点了断开 → 切其他应用 → 切回 YueLink，过去会出现 UI 显示节点名 + 连接图标但实际无网的情况。根因是 `userStoppedProvider` 只活在内存里，引擎重建（Android 后台杀引擎）会重置回 false，恢复路径就把 UI 拉回 running。新版把"用户主动停"持久化到磁盘，冷启动 + resume 都先读持久化标记。**含一个回归向修复**：开了 auto-connect 的用户，断开后杀 app 重开也不会被误启动，但手动 connect 之后切应用回来仍然正常工作。(`a35db1f`)

- **v2rayN / vr2 等代理客户端劫持系统代理后能立即恢复**：之前 YueLink 的 system proxy 校验缓存 60 秒，window focus / resume 拿不到强制刷新入口，意味着用户在桌面上打开 v2rayN 干扰了系统代理设置后，最长要等 60 秒 + 30 秒心跳才会恢复，期间一直显示「已连接」但实际无网。新版 `verify(force: true)` 绕过缓存；`set()` / `clear()` 后所有平台都立即清缓存；resume 时如果是 systemProxy 模式，直接强校验 + 必要时 restore，不等 30 秒心跳。(`43ccad9`)

- **断开重连后立即测速不再"全红"**：v1.0.20 已经修了"`testGroupDelay()` 返回全超时"那条路径，让它走 flush+retry。但漏了对称的另一条：`testGroupDelay()` 自身抛异常（HTTP 超时 / mihomo 5xx）时直接把节点全标红，跳过 recovery。这恰好是重连后最常见的失败形态——mihomo 内部状态没完全 reset 时，第一个请求经常根本不返回 result map。新版把两种失败都统一走同一个 flush+retry helper，并在 telemetry 里加 `reason` 字段（`all_timeout` / `exception`）方便区分。(`eecd12d`)

- **mihomo 吞吐对齐上游（额外 ~30%）**：v1.0.20 已在客户端 `_ensureSniffer` 里 strip 掉 `parse-pure-ip` 和 `force-dns-mapping` 两项 throughput-killing flag，但 `assets/default_config.yaml` 里还遗留旧值，订阅退化到 fallback 模板时会带上。本次客户端把 asset 也清干净；同时**配套的服务端模板**也已对齐（`/home/xboard/yue-to/rules/{default,app}.clash.yaml`）—— 后者额外修复了 `store-fake-ip: true` 这个长期隐性 bug：每日订阅更新会让 fake-IP 缓存与新规则冲突，部分网站打不开除非用户清 `cache.db`。**这条是今天最大杠杆的修复**，受益面包括所有用此面板的客户端（不限 YueLink）。(`ccfae5e`)

- **Windows 退出彻底**：v1.0.20 加的 3 秒 `Process.killPid` 兜底用 Dart `Future.delayed` 调度——和待清理工作排在同一个事件循环里。如果某个 native 调用（windowManager.destroy / ServiceClient.stop）卡住了主 isolate，3 秒计时器就和它一起被卡住，结果托盘点 Quit 之后窗口关了但 yuelink.exe 仍在 Task Manager。新版把 watchdog 移到独立 isolate，事件循环互不干扰，主 isolate 卡住 watchdog 仍能 fire SIGKILL/TerminateProcess。(`24965f8`)

- **托盘菜单加"模式"快切**：托盘右键 → 模式 → 规则 / 全局 / 直连 / 系统代理 / TUN，✓ 标记当前选中。逻辑共享 `ModeActions` —— 主页 pill 点击和托盘点击走同一份代码，不会出现"pill 显示一个、托盘显示另一个"的不同步状态。(`70c4e31`、`b8d8c6a`)

- **自动恢复连接不再静默闪烁**：心跳累计 3 次失败后会触发"silent restart"，在 v1.0.20 里完全无感，从用户视角看就是"刚才连着突然断了一秒又连上"，看着像应用乱跳。新版加三段透明提示：`正在自动恢复连接...` → `已自动恢复连接` → 如果两轮都失败，`自动恢复失败，连接已断开`。中间所有状态都写 EventLog，导出诊断时可还原全过程。(`29ed80f`)

### 修复

- **桌面 TUN / systemProxy 切换**：之前桌面热切换只走 `mihomo PATCH /configs`，但 TUN 必须经过 service helper（特权助手），systemProxy 在 app 进程内即可。PATCH 跨不了这个进程边界，结果 UI 显示 TUN 但 runtime 还是 systemProxy。新版桌面切换走完整 restart：失败时回滚到原 mode 并尝试再启动一次，避免直接把用户丢到断网状态。同步给"安装服务模式"按钮加 helper readiness 等待（最长 12s）+ 3 次启动重试（0/2/4s 节奏），降低首次安装时的 race。安装/更新服务这个动作本身被解读为"用户要 TUN 了"，会自动清掉 P0-1 的 manualStopped 标记，避免装好了不自动起。(`7de57f8`)

- **路由模式切换的 verify 校验恢复**：`ModeActions.setRoutingMode()` PATCH 成功后会再读一次 mihomo 的实际模式，如果发现订阅强行覆盖了我们的请求，给 warning 提示而不是无声 success。(`b8d8c6a`)

### 服务端配套（无需用户操作）

XBoard 订阅模板 `/home/xboard/yue-to/rules/{default,app}.clash.yaml` 已同步对齐：

- 删除 `parse-pure-ip: true` 行
- `store-fake-ip: true` → `store-fake-ip: false`

两项都跟客户端 `ConfigTemplate` 的"single source of truth"保持一致。所有用此面板的客户端（不限 YueLink）下次同步订阅就拿到新模板。

### 升级提示

- **已登录用户**无需重登，订阅与设置自动保留。
- **吞吐 / cache.db 修复**通过服务端模板对齐生效，下次同步订阅自动拿到，**对客户端版本本身无强制升级要求**——但只有 1.0.21+ 才同时享有上面其他 8 项修复。
- **断开 / 自动恢复 / 系统代理校验**类修复 100% 客户端侧，必须升级到 1.0.21 才能拿到。
- **Windows 退出 / 桌面 TUN 切换**类修复同上。

### 已知问题

- `main.dart` 内 5 条 `unnecessary_underscores` analyze info 仍在，全部历史代码，无功能影响。
- Windows / macOS / Android 部分手测路径无 CI 自动化覆盖（`CoreLifecycleManager` / `CoreHeartbeatManager` 等的 wire-up 部分），靠 commit body 中列的手测清单做 release gate。
- v1.0.20 已知问题（`SecureStorageMigration` 未启用、`win32 5 → 6` 联动延后、`CHANGELOG.md` 历史欠账）本次 hotfix 范围内不动，沿用 v1.0.20 状态。
