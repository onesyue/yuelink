## Fixes

- **修复 DNS 泄漏到中国 DoH** — 之前 TUN 模式下访问国外网站时 DNS 查询会被 AliDNS / TencentDNS 看到。`respect-rules: true` 控制的是「DNS 查询包从哪条路出去」，**不是「问哪台 DNS 服务器」**；订阅默认 `nameserver` 是国内 DoH，非 CN 域名 fall through 到这里就泄漏了。
  - **服务端订阅模板**：`nameserver-policy` 加 `geosite:geolocation-!cn` catch-all → Cloudflare DoH / Google DoH。mihomo 优先级 specific-suffix > geosite，已有的 AI 域名 / Apple specific 规则继续生效。`fake-ip-filter` 末尾加 `geosite:cn`（OpenClash 标准做法），让整个中国 geosite 数据集（约 10 万域名）直接真实 DNS 解析、不走 fake-IP，国内 CDN 调度更准、首屏更快。
  - **客户端兜底**：`dns_transformer.ensureDns` 三处主动注入 `geosite:geolocation-!cn` policy（订阅没带时 client 加，覆盖 flow `{}` 和 block 两种 YAML 形态）。配合服务端，国外域名查询不会落到 CN DoH。

- **改善节点池稳定性 — 死节点自动跳过** — 之前 11 个国家聚合（HK / TW / JP / KR / SG / US / UK / DE / NL / AU / CA）使用 `load-balance + consistent-hashing`，没有 `expected-status`，节点 TCP 连通就算"健康"。结果某些节点 TLS 握手失败 / 被服务端 reset / 被 Cloudflare 反代理 403 时，仍然被分流，连接不稳。mihomo 源码 `outboundgroup/loadbalance.go:52` 显示 consistent-hashing 按 eTLD+1 哈希、不真 sticky（`chat.openai.com` 和 `chatgpt.com` 哈希到不同节点），没法靠它给 ChatGPT 提供 IP 亲和性。
  - **服务端订阅模板**：11 个国家聚合从 `load-balance + consistent-hashing` 改 `url-test + tolerance: 50 + expected-status: 200-299`。`expected-status` 让 cf-WAF 拒（403）/ 5xx 也算"不健康"，节点会被自动踢出聚合，下次选最快的活节点。

- **TUN 模式下流媒体偶发被错路由到地区组** — Netflix / Disney+ / 部分 AWS-fronted 服务的 CDN edge IP 在不同时段会漂移，mihomo 默认 sniffer 在某些短路径下会跳过 SNI sniff，直接按 GeoIP 路由（IDC 归属国家），结果用户的"Netflix"流量被分到"美国聚合"而不是"流媒体"组，触发节点策略错位。
  - **客户端 + 服务端订阅模板**：`sniffer.force-domain` 列表新增 `+.netflix.com` / `+.nflxvideo.net` / `+.nflximg.com` / `+.nflxext.com` / `+.nflxso.net` / `+.amazonaws.com` / `+.disney-plus.net` / `+.dssott.com` / `+.media.dssott.com`（OpenClash 路由器主流默认列表）。这些域名总是触发 SNI sniff → 按域名路由到流媒体组。

- **优化 TCP 长连接稳定性** — 之前 `keep-alive-interval: 30s` 没配套 `keep-alive-idle`，mihomo 落到 OS 默认（Linux / macOS / Windows 都是 7200 秒 = 2 小时）才发第一个 keep-alive 探测包。等于现有的 30s interval 形同虚设，挂掉的 NAT 绑定被发现得晚、用户体感是"代理用着用着突然卡 1 分钟"。
  - **客户端**：加 `keep-alive-idle: 600`（10 分钟）。空闲 10 分钟后开始每 30 秒发探测，匹配 Clash Verge Rev / mihomo-party 主流默认。短到能及时发现 NAT 失效，长到不会唤醒移动设备的无线模块。

- **加速 Apple Push / iCloud 等 fake-IP 长连接重连** — `profile.store-fake-ip` 之前是 `false`，每次 yuelink 重启（模式切换 / 订阅同步 / 内核重载）都重建空 fake-IP 表。OS 级 DNS 缓存还留着重启前的 fake-IP，这些请求进 mihomo 后命中"fake DNS record missing"分支，靠 sniffer 救场拿 SNI，多一道弯，前几秒首包延迟可见。
  - **客户端**：`profile.store-fake-ip: true`。fake-IP 表跨重启持久化，重启后 OS DNS 缓存的 fake-IP 还能正确反查到原域名。Clash Verge Rev / mihomo-party / OpenClash 都用 `true`。`fake-ip-filter` 改变时 mihomo 自动 invalidate 不再匹配的 entry，没有 stale 风险。

## Internal

- **设计审查**：基于 OpenClash 路由器实地配置 + 12 个 mihomo 顶层字段的源码级核实（`core/mihomo/config/config.go:480-583`、`utls.go:42-78`、`reality.go:39`、`dispatcher.go:262-275`），系统性识别 yuelink 当前缺失 / 应保持的设置。最终决策：

  | 字段 | 决策 | 理由 |
  |---|---|---|
  | `global-client-fingerprint: random` | KEEP（不加）| `OnceValue` 进程级 sticky，**不真 anti-fingerprint**；REALITY 自带 `chrome` 强制覆盖 |
  | `keep-alive-interval` | KEEP 30s | 在 carrier NAT 阈值（~60s）下面，CPU 唤醒减半 |
  | `keep-alive-idle` | **新加 600s** | 之前没设 → OS 默认 7200s 让 interval 形同虚设 |
  | `experimental.quic-go-disable-gso` | KEEP（不加）| 仅 OpenWrt + Realtek/MTK NIC 驱动 bug，桌面/移动正常 |
  | `tcp-concurrent` / `unified-delay` / `profile.store-selected` | KEEP 已有 | 全套 2026 客户端通用 |
  | `profile.store-fake-ip` | **改 true** | 重启后 fake-IP 表保留，OS DNS 缓存能正确反查 |
  | `ntp` 块 | KEEP（不加）| 桌面 OS 自己同步够 |
  | `find-process-mode: always` (desktop) | KEEP 已有 | 订阅有 PROCESS-NAME 规则 |
  | `sniffer.parse-pure-ip / force-dns-mapping` | **继续 strip** | v1.0.21 P1-4 测出 30% 吞吐量回归，`dispatcher.go:262-275` 上游没修过 |

- **服务端订阅模板基础设施固化**：`/home/xboard/yue-to/xboard-templates/` 独立 git repo + commit 当前快照 + README 说明数据流（git → patch-subscribe-templates.sh → DB v2_subscribe_templates → ClashMeta.php 渲染），即便上游 cedar2025/Xboard 镜像更新或 git pull 也不影响订阅模板。`/home/xboard/yue-to/rules/` 历史遗留目录（早期 XBoard 版本读 `/www/resources/rules/*.yaml`，新版改 DB）已确认不被使用，从 docker compose mount 中移除并删除。
