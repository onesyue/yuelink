# Server file inventory + redeploy SOP

Single source of truth for **what lives where, how it survives upgrades,
and how to rebuild from scratch on a fresh host**. Written 2026-05-06
after a session that closed several hand-deployed gaps. Update this file
whenever a new file leaves the git tree.

The `23.80.91.14:/opt/yueops/` directory is itself a git checkout of
this repo. **Pulling new commits is `cd /opt/yueops && git pull`**
on a writable deploy key — the deploy key currently installed there
is read-only, so push has to happen from a workstation, then `git
pull` from the host. Local edits on the host should be considered
ephemeral and re-pushed via your workstation, not committed in place.

## Restore-from-scratch SOP

Goal: rebuild a host fully from `(this repo) + (DB backup) + (credentials)`.

### Restoring `23.80.91.14`

1. Provision Debian 13 VM, install `python3 python3-venv git nginx
   sshpass postgresql-client dnsutils sysstat`.
2. Clone yueops: `git clone https://github.com/onesyue/yueops /opt/yueops`.
3. Clone yuelink: `git clone https://github.com/onesyue/yuelink /tmp/yuelink-repo`.
4. Build checkin-api:
   ```
   mkdir -p /opt/checkin-api && cd /opt/checkin-api
   cp /tmp/yuelink-repo/server/checkin-api/* .
   cp /tmp/yuelink-repo/server/telemetry/telemetry.py .
   cp /tmp/yuelink-repo/server/telemetry/dashboard.html .
   python3 -m venv venv && venv/bin/pip install -r requirements.txt
   ```
5. Recreate `/etc/systemd/system/checkin-api.service` (template inline
   in `server/checkin-api/README.md`) — fill in env vars from the
   secrets vault.
6. Recreate `/etc/systemd/system/yueops-dns-scheduler.{service,timer}`.
   The `.service` template is inline in
   `scripts/dns-scheduler/scheduler.py` header.
7. `systemctl daemon-reload && systemctl enable --now checkin-api yueops-dns-scheduler.timer`.
8. Restore `cf_dns_*` credentials into yueops `settings` table (or
   import via yueops admin web UI).

### Restoring `66.55.76.208`

1. Provision, install docker.
2. Clone yuelink: `git clone https://github.com/onesyue/yuelink /tmp/yuelink-repo`.
3. nginx:
   ```
   sudo mkdir -p /home/nginx/{conf.d,conf,certs,html,logs}
   sudo cp /tmp/yuelink-repo/server/nginx/*.conf  /home/nginx/conf.d/
   sudo cp /tmp/yuelink-repo/server/nginx/favicon.ico  /home/nginx/html/
   sudo cp /tmp/yuelink-repo/server/nginx/icon.png     /home/nginx/html/
   # nginx.conf and certs are not in repo — restore from backup or LE re-issue
   docker run -d --name nginx --restart=always -p 80:80 -p 443:443 \
       -v /home/nginx/conf.d:/etc/nginx/conf.d \
       -v /home/nginx/conf/nginx.conf:/etc/nginx/nginx.conf \
       -v /home/nginx/certs:/etc/nginx/certs \
       -v /home/nginx/html:/usr/share/nginx/html \
       -v /home/nginx/logs:/var/log/nginx \
       nginx:1.27-alpine
   ```
4. XBoard (refer to ghcr.io/cedar2025/xboard image's own deploy
   guide — that side isn't owned by us). Restore `/home/xboard/yue-to/`
   tree from backup; the bind-mounted dirs include all our customisations.
5. Re-run any container-replay scripts that aren't bind-mounted, in
   particular:
   ```
   /home/xboard/yue-to/patch-loon-upload-bandwidth.sh
   ```
6. Restore Postgres data (`yueops` db).

## Upgrade SOP (vs full rebuild)

Most days you only want to upgrade one piece without touching others.

### Upgrade XBoard container

```bash
docker pull ghcr.io/cedar2025/xboard:latest
docker compose up -d   # the compose file in /home/xboard/yue-to/ rebuilds yue-to-web-1
# After it comes up:
bash /home/xboard/yue-to/patch-loon-upload-bandwidth.sh   # idempotent
docker exec yue-to-web-1 php /www/artisan cache:clear
```

The bind-mounted `/home/xboard/yue-to/rules/` already contains the
ECH/DoH fix from 2026-05-06 — no further action needed.

### Upgrade nginx container

```bash
docker pull nginx:1.27-alpine
docker rm -f nginx
docker run ...   # same args as in restore steps
```

All host-side bind-mounts (favicon, default.conf, etc.) survive.

### Pull a new yueops scheduler change

On `23.80.91.14`:

```bash
cd /opt/yueops && git pull
# scheduler.py is reloaded on the next 30-second tick automatically
# (oneshot service, exits and is re-spawned by the timer).
```

### Pull a new checkin-api / telemetry change

On a workstation where you can scp:

```bash
cd <yuelink repo on workstation>
git pull
sshpass -e scp server/checkin-api/main.py        root@23.80.91.14:/opt/checkin-api/
sshpass -e scp server/checkin-api/web_gateway.py root@23.80.91.14:/opt/checkin-api/
sshpass -e scp server/checkin-api/nodes-inventory-path-map.json root@23.80.91.14:/opt/checkin-api/
sshpass -e scp server/telemetry/telemetry.py     root@23.80.91.14:/opt/checkin-api/
sshpass -e scp server/telemetry/dashboard.html   root@23.80.91.14:/opt/checkin-api/
sshpass -e ssh root@23.80.91.14 "systemctl restart checkin-api"
```

A future improvement is to give `23.80.91.14` a writable git remote
for yuelink and let it `git pull && cp -r server/{checkin-api,telemetry}/* /opt/checkin-api/`
locally. Not done today.

## Idempotency markers

Every patch we ship to a hand-deployed file carries a comment marker
so re-running a patch script doesn't double-apply. Greppable list:

| Marker | File | Owner |
|---|---|---|
| `# yuelink:cymru-v6+carrier` | `server/checkin-api/main.py` | 2026-05-06 |
| `# yuelink:cymru-retry-no-failure-cache` | `server/checkin-api/main.py` | 2026-05-06 |
| `# yuelink:secure-dns-routing` | injected runtime into subscription rules: by `RulesTransformer.ensureBrowserSecureDnsRules` | 2026-05-06 |

Anything else with `# yuelink:` is fair game to add — pick a
descriptive suffix.

## When to extend this file

Add a row whenever:

- A new file gets hand-deployed (not in any git repo, not bind-mounted).
- A bind-mount is added or removed.
- A backup table is added or dropped.
- A new credential vault location appears.

The goal is that anyone restoring from scratch can read this one file
and know exactly which paths to populate. If a step had to be inferred
from chat history or memory, that's a gap.
